## 文体・トーン

敬体ベースは維持する。問題は敬体か常体かではなく、**文末と段落の均質さ**（AI臭の正体）である。次の6ルールに従う。

1. **敬体ベース＋文末の多様化**
   - です・ます基調、平易な日本語（中学生〜社会人の初心者が読めるレベル）
   - 同一文末（〜です／〜ます）を**3連続させない**
   - **体言止め**は段落の区切り・山場に限って使う。ある程度の長さがある記事では**ゼロ回避**（少なくとも1回）。**文比率の目標は設けない**（45〜75%／60%±5% 等は不採用）
   - 短い常体の断定は山場のみ混在可（例: 「特別な仕掛けはなし。」）
   - 一文の長さを意図的にばらす（短→長→短の拍）
2. **一人称は必要時だけ＋実在感要件**
   - 通常は主語を省略し、**「筆者」も連発しない**。「筆者」は実測・失敗談・確認範囲・未検証の境界を示す必要がある文だけに使う
   - 「筆者」を使う場合、同じ文か段落で使用理由（実測した／失敗した／確認範囲を限定する等）が読者に伝わるようにする
   - 「私」「当サイト」は使わない
   - 記事ごとに最低1つ、内部チェック用の一次情報マーカーを置く: **実測数値＋取得日**／**使用環境**／**失敗談**のいずれか
   - 「一次情報」「筆者の確認範囲」は内部の判定語。公開見出しには原則出さず、「実際に試して分かったこと」「試した環境と結果」など読者の関心に合わせる
   - 実測を持たないテーマは捏造せず、確認日・参照元・未検証範囲を具体的に書く
3. **断定と逡巡の使い分け**
   - 実測で確定したことは言い切る。未確定は「今回の環境では」「2026年7月時点で確認した範囲では」など、主語を増やさず範囲を限定する
   - **結論の先延ばしは禁止**（「今日の結論」は維持）
4. **LLM空句の禁止** — 正本は [`llm-forbidden-phrases.md`](./llm-forbidden-phrases.md)。「最後に」「まさに」「体言止め」は禁止しない
5. **段落設計**
   - 1段落1トピック＋密度波形（密な段落2〜3個ごとに疎を1つ）
   - 「状況→理由→対策」の同一テンプレ連打を避ける
6. **因果に機構を一文添える** — 結果だけでなく、なぜそうなるかの仕組みを一文入れる

口語的なつなぎは可（「正直」「実際に」）が、冗長な感嘆は避ける。英語用語は初出で日本語を添える（例: token（トークン））。推測は「〜と考えられる」「〜の可能性」と書く。

### 句読点

- 文末が丸括弧で終わる場合は `（…）` で閉じ、`（…）。` と句点を重ねない
- 括弧の後にも本文が続く場合は、文全体の終わりに通常どおり句点を置く

## 記事の型

### 推奨構成

1. 導入（読者の悩みに共感、1〜3段落）
2. **今日の結論**（`> **今日の結論**` または `> **この記事の結論**` の引用ブロック、箇条書き 3〜5 点）
3. 本論（H2 単位で問題→原因→対策）
4. チェックリスト or 表（あるとよい）
5. まとめ / 次に読む
6. 免責（仕様変更・執筆時点の注記）

### 記事タイプ別の推奨導線

| タイプ         | 推奨導線                                                              |
| -------------- | --------------------------------------------------------------------- |
| How-to         | 手順・チェックリストで無料価値 → 記事末にデジタル商品 CTA             |
| 比較・選定     | 比較表 + 向く人/向かない人 → アフィリエイト（判断材料をリンク前後に） |
| チェックリスト | 本文で完結する価値 → テーマに合う CTA を1本に絞る                     |
| Hub            | スポークへの内部リンクを優先 → 必要に応じてシリーズ関連 CTA           |

### 見出し

- **H1 は記事タイトル1つのみ**（本文中に H1 を増やさない）
- **H2 は最大 5 前後**を目安（長文はシリーズ分割を優先。公開済み長文で超えることはある）
- H3 は H2 の下で論点を分けるときのみ
- 見出しだけを追っても「何が分かるか／何をするか」が推測できる情報量にする。「判断基準」「一次情報」「FAQ」など抽象ラベル単独は避ける
- 長文化させず、結論・対象・数字のいずれか1つを足す（例: 「判断基準」→「少人数なら先に埋める3つの軸」）
- 見出しにキーワードを無理に詰め込まない（自然な日本語優先）

### 箇条書き・表

- 箇条書きは **1項目 1〜2 行**、並列の粒度を揃える
- 比較・可否は **表** を優先（Markdown 表）
- 手順は番号付きリスト

## Markdown ルール

- コードブロックに言語指定（`md`, `text`, `bash` など）
- 画像: `![説明](ファイル名.png)` — 説明文は具体的に（`captions.md` があればそれに合わせる）
- 内部リンク: `/blog/slug` 形式（slug は `lib/blog/posts.ts` のキー。未公開は「準備中」と書く）
- 水平線 `---` は大セクションの区切りに使う（乱用しない）
- 強調: 太字は**要点の語句のみ**（段落全体を太字にしない）

## Frontmatter（記事ドラフト用）

記事として出力するときは、先頭に YAML を付ける:

```yaml
---
title: "32文字前後で検索意図が分かるタイトル"
description: "120〜160文字程度。誰向け・何が分かるか・実測などの差別化を1文で"
date: YYYY-MM-DD
tags:
  - タグ1
  - タグ2
site: toolarc.jp
target: "読者像を1行で"
last_update: YYYY-MM-DD
---
```

### Frontmatter の補足

- `title` と本文先頭の `#` 見出しは**意味を揃える**（完全一致でなくてよいが、検索意図は一致させる）
- `tags` は 5〜10 個程度。記事テーマ・ツール名・シリーズ名を含める
- `site` は `toolarc.jp`（引用符の有無は既存記事に合わせてよい）
- `last_update` は更新したときだけ追記（任意）

## 収益導線（本文での書き方）

`AGENTS.md` の Monetization Rules の実務版。本文の信頼を損なう導線は入れない。

### CTA 配置（収益・CV記事）

第2〜3層の収益記事・本命CVでは、**同一の主目的／遷移先**を次の3点に置く（別商品の並列表記ではない）:

1. リード直後（今日の結論付近）
2. 本文中（比較表直後など）
3. まとめ前

- CV文言は**無料オファー型**を優先する（「詳細はこちら」のみにしない）
- 本文の主力は**ボタン型テキストリンク**
- **バナーは記事末・サイド等の面のみ**（本文中盤へのバナー連打は禁止。比較記事本文へのバナー直貼り禁止）

Tips／チェックリストなど非収益型は、従来どおり**テーマに合う CTA を1本に絞る**（上記3点は収益記事向け）。

### アフィリエイト

- 比較・選定記事では **表** と「向く人 / 向かない人」をセットで書く
- リンクの前後に判断材料を置く（注意点、代替案、費用感、前提条件）
- 機能羅列ではなく、読者の課題解決に紐づけて訴求する
- 「絶対」「必ず」などの断定は禁止。条件・前提を明記する
- 直アフィ禁止の比較シリーズは、周辺収益記事への弱リンクのみ（`lib/affiliate/policy.ts` / 運用メモに従う）

### 広告

- 記事品質・滞在価値・情報密度を優先する（広告収益のために本文を薄くしない）
- 過剰な挿入や本文の分断は避ける
- 広告は補助収益。まずは SEO と読者満足を満たす

### デジタル商品（note / 教材 / テンプレ）

- 記事内で先に無料価値を提供する（手順、チェックリスト、判断基準）
- **誰向けかが明確な CTA** を置く。非収益型・Tips は記事末に1つ（複数の別商品を並べない）。収益記事は上記「CTA 配置」の3点に同一オファーを置く
- 本文テーマと無関係な商品誘導はしない

## SEO

- **検索意図**: 初心者が今困っていること（How-to、チェックリスト、比較）
- **タイトル**: 主要キーワード + 読者メリット（煽り・過剰な記号は避ける）
- **description**: 誰向けか・何が分かるか・実測・差別化を1文に集約
- **本文**: キーワードの詰め込みより、見出しと表で意図を満たす
- **Hub 記事**: シリーズ全体の入口として、スポーク記事への内部リンクを入れる

---

## 禁止表現・注意

- 根拠のない「絶対」「必ず」「必ず成功」「公式保証」
- 景表法に抵触しうる誇大表現・過度な煽り・クリックベイト
- 未確認の料金・制限・移行可否の断定（**執筆時点の日付**を本文または免責に書く）
- 公式保証がない内容を「公式」と表現しない（公式ドキュメントで確認できる場合のみ「公式によると」等）
- OpenAI / Anthropic / Cursor 等の未確認仕様を事実として断定しない
- Export ZIP / conversations.json を「そのまま AI に渡せば再現できる」と書かない
- 会話ログの羅列を記事本文の主役にしない
- daily notes の丸投げ記事化
- ダーク SaaS / サイバーパンク風の UI・デザイン提案

---

## 仕様・免責（記事末尾）

次のいずれかに近い形で締める:

- OpenAI / Anthropic / Cursor 等の仕様は変更されうる旨
- 記事は執筆時点（`date` / `last_update`）の情報に基づく旨
- 重要な判断は公式ドキュメントで確認する旨
