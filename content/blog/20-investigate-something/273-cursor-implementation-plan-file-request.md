# CursorのPlan Modeでplan.mdが作られないときの依頼文

Cursor の Plan Mode で、以前のように `plan.md` が作られなくなったと感じたことはないでしょうか。

筆者の環境では、2026年6月18日頃から Plan Mode の挙動が変わったように見えました。以前は、実装前に「どのファイルをどう変更するか」をまとめた `plan.md` が作成されることがありました。しかし最近は、`plan.md` が生成されず、Ask Mode で実行するか Agent Mode に切り替えて実行する流れになる場面がありました。

この記事では、これまで通り **計画ファイルを確認してから Agent Mode で実装したい** ときの依頼文をまとめます。

> **今日の結論**
>
> - 以前の Plan Mode では、実装前に変更方針をまとめた `plan.md` が作成されることがあった
> - `plan.md` は変更内容の確認だけでなく、後から実装意図を見直す記録として便利
> - 2026年6月18日頃から、筆者環境では Plan Mode だけでは `plan.md` が作られない場面が出た
> - これまで通り計画ファイルを残したい場合は、依頼文に「`plan.md` を作成してから実装」と明記する
> - Cursor の仕様は変わる可能性があるため、Plan Mode で `plan.md` が出ない場合の暫定対策として使う

---

## 以前のPlan Modeではplan.mdが作られていた

以前の Plan Mode では、実装に入る前に「どのファイルを、どの順番で、どう変更するか」を整理した `plan.md` が作成されることがありました。

このファイルがあると、実装前に変更方針を確認できます。たとえば「このファイルは触ってよいが、別のファイルは触らないでほしい」といった判断を、実装に入る前に差し戻せます。

また、後から見直したときにも便利です。コード差分だけを見ると「何が変わったか」は分かりますが、「なぜその変更にしたのか」は分かりにくいことがあります。`plan.md` が残っていれば、当時の判断理由を追いやすくなります。

---

## plan.mdがあると何が便利なのか

実装依頼の結果だけを見ると、コードがどう変わったかは分かっても、次のような経緯が分かりにくくなります。

- なぜその実装方法を選んだのか
- 他に検討した選択肢があったのか
- どこまでが依頼の範囲で、どこからが Cursor の判断だったのか
- 実装前に確認すべき注意点が何だったのか

特に複数回に分けて実装を進めている場合、後から見返すときの手がかりが少なくなってしまいます。

`plan.md` を DailyNote や AI-log と一緒に残しておくと、次に同じ機能を直すときに「前回どこまで考えたか」を思い出しやすくなります。実装差分と計画をセットで残すのがポイントです。

---

## 2026年6月18日頃から挙動が変わった

筆者の環境では、2026年6月18日頃から Plan Mode の挙動が変わったように見えました。

以前のように Plan Mode で `plan.md` が作られるのではなく、次のどちらかを選ぶ流れになる場面がありました。

- Ask Mode で実行する
- Agent Mode に切り替えて実行する

このため、以前のように「計画ファイルを確認してから Agent Mode で実装する」流れを、そのまま再現しにくくなりました。

もちろん、これは筆者の環境での実測です。Cursor の UI や Plan Mode の仕様は変わる可能性があり、すべての環境で同じ挙動になるとは限りません。ただ、同じように `plan.md` が作られず困っている場合は、次の依頼文を試す価値があります。

---

## plan.mdを残したいときの依頼文

これまで通り計画ファイルを残したい場合は、依頼文の中で `plan.md` の作成を明示します。

```text
実装前に、変更方針を plan.md として作成してください。
plan.md には、対象ファイル・変更内容・実装手順・確認項目をまとめてください。

その後、内容を確認してから Agent Mode で実装します。
```

実際の実装依頼に組み込むなら、次のように書くと使いやすいです。

```text
〇〇機能を実装したいです。
対象は src/lib/xxx.ts と app/xxx/page.tsx です。

まず plan.md を作成し、変更対象ファイル・実装方針・確認項目を整理してください。
すぐに実装せず、plan.md の内容を確認してから Agent Mode で進めます。
```

ポイントは、単に「Plan Modeでお願いします」と書くのではなく、**何を plan.md に書いてほしいか**まで指定することです。対象ファイル・変更内容・実装手順・確認項目の4つが入っていれば、後から見返しても使いやすい記録になります。

---

## うまくいかない場合の確認ポイント

`plan.md` が作られない場合は、依頼文と進め方を確認します。

| 確認項目 | 内容 |
| --- | --- |
| 依頼文 | `plan.mdを作成してください` と明記したか |
| 実装前停止 | すぐ実装せず、計画確認を挟むよう指示したか |
| 記載内容 | 対象ファイル・変更内容・実装手順・確認項目を指定したか |
| ファイル名 | `plan.md` として残したいのか、Cursor側の計画ファイルでよいのか確認したか |
| 記録 | `plan.md` と実装差分を DailyNote / AI-log に残したか |

特に「すぐに実装せず、先に計画を出してください」と書くことが重要です。実装まで一気に依頼すると、計画がチャット内の説明だけで終わり、ファイルとして残らない場合があります。

---

## まとめ・次に読む

Plan Mode の挙動は今後も変わる可能性があります。ただ、実装前に計画を残す価値は変わりません。

もし Plan Mode で以前のように `plan.md` が作られない場合は、依頼文に「`plan.md` を作成してから実装」と明記してみてください。実装前の確認にも、後日の振り返りにも使いやすくなります。
