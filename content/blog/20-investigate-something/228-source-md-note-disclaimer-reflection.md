# source.mdのnote欄「要確認」を記事免責へ反映する

> **この記事の結論**
>
> - source.md の note 欄は、執筆時点の不確実性や前提条件を書き残す場所として使える
> - note に「要確認」と書いた内容は、Claude 初稿の**免責文**と**本文の条件付き表現**に反映させる
> - 未確認の仕様を断定しない運用と組み合わせると、記事の信頼性を保ちやすくなる

---

## なぜ note 欄に「要確認」を書くのか

Claude に記事初稿を依頼するとき、source.md には「伝えたいこと」や「想定読者」だけでなく、**執筆時点でまだ確認しきれていない情報**が混在していることがあります。

その不確実性を曖昧なまま初稿に渡すと、Claude が断定表現で書いてしまうことがあります。「〇〇は必ずこうなる」「〇〇で確実に改善できる」といった表現は、仕様変更や個別環境の違いによって誤りになりかねません。

note 欄への「要確認」メモは、**そのリスクを初稿生成の段階から防ぐ**ための仕組みです。筆者は [GSC記事は表示文言の変更可能性を免責に明記する](/blog/gsc-article-ui-change-disclaimer) の執筆時にこの運用を試し、断定表現の回避と組み合わせて使えることを確認しています。

---

## 手順：note 欄から免責へ反映する流れ

### 1. source.md 作成時に「要確認」を note に書く

記事を設計する段階で、不確実な点や前提条件を note 欄にメモします。

```md
note: GSC表示文言は変更される可能性あり（要確認）
note: 無料プランでの動作は未確認（要確認）
```

複数ある場合は箇条書きで列挙しておくと、初稿依頼時にそのまま貼り付けられます。

### 2. Claude 依頼文に反映指示を明記する

初稿を依頼するプロンプトに、note の内容を免責・本文へ反映する指示を一行添えます。

```md
<!-- 依頼文への追記例 -->
source.md の note にある「要確認」事項を、
本文の条件付き表現と末尾の免責に反映してください。
```

この一行があるだけで、Claude は note の内容を「断定せず、条件をつけて書く」指針として扱います。

### 3. 初稿の断定表現をチェックする

初稿が上がったら、次の表現が残っていないか確認します。

| 断定表現（NG） | 条件付き表現（OK） |
| --- | --- |
| 必ず改善できます | 改善できる可能性があります |
| 絶対に表示されます | 執筆時点では表示されます |
| 公式の仕様です | 執筆時点の動作です |

表現を修正する際は、「〜の可能性があります」「執筆時点では〜です」に置き換えるのが基本です。

### 4. 末尾免責に「執筆時点」の注記を入れる

記事末尾には、定型の免責文を置きます。

```md
本記事は 20XX-XX-XX 時点の情報をもとに執筆しています。
仕様・UI・料金は変更される可能性があります。
最新情報は各サービスの公式ドキュメントでご確認ください。
```

note の内容が具体的な場合（GSC の表示文言、特定ツールのプラン制限など）は、該当箇所に合わせた一文を加えます。

### 5. 公開後に note を更新する

公開後、note に書いた「要確認」事項が確認できた場合は、note を更新してメモを残します。仕様が変わっていた場合はリライト候補としてフラグを立てておくと、更新作業が漏れにくくなります。
