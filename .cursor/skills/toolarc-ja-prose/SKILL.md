# toolarc-ja-prose（ToolArc日本語推敲）

ToolArc記事の意味・検索意図・導線を維持しながら、日本語の機械的な均質さを減らす推敲手順。

判定基準を本Skillへ複製しない。文体・SEOの正本は `docs/ai-context/writing-rules.md`、LLM空句の正本は `docs/ai-context/llm-forbidden-phrases.md`。矛盾時は正本を優先する。

## 対象 / 対象外

**使う**:

- 記事初稿の日本語推敲
- 既存記事のAI臭・読み味・リズム改善
- 文末、段落、見出し、根拠表現の局所修正

**使わない**:

- Output Contractの合否判定（`l1-review-article`）
- 収益構成・CTA設計（`revenue-article-template`）
- `posts.ts`登録、公開日確定、build（`publish-article`）
- 検索意図、事実、料金、仕様、実測値の創作

## 手順

### 1. 守る内容を固定する

対象記事の検索意図、結論、事実、内部リンク、CTA、免責を確認する。依頼が文体改善だけなら、構成変更や情報追加へ広げない。

### 2. 6ルールで推敲する

1. **敬体ベース＋文末を散らす**
   - 同一文末（です／ます）を3連続させない。
   - 文長を短→長→短など意図的にばらす。
   - 体言止めは段落の区切り・山場だけに使い、ある程度の長さがある記事ではゼロを避ける。比率目標は置かない。
2. **一人称を必要時だけ使う**
   - 通常は主語を省略する。「筆者」は実測・失敗談・確認範囲・未検証の境界を示す場合だけ使い、使用理由が同じ文か段落で分かるようにする。
   - 実在感は代名詞ではなく、実測数値＋取得日、使用環境、失敗談で示す。無い情報は作らない。
3. **断定と逡巡を分ける**
   - 実測で確定したことは言い切る。
   - 未確定事項は「今回の環境では」「確認した範囲では」など、範囲を具体化する。
   - 結論を遅らせない。
4. **空句を削る**
   - `llm-forbidden-phrases.md`を確認し、意味のない強調、実況、曖昧な網羅語、接続詞の連打を削除または具体化する。
   - 単語の一律禁止にせず、文中で情報が増えているかで判断する。
5. **拍と段落密度を変える**
   - 1段落1トピックを守る。
   - 密な段落2〜3個ごとに疎な段落を置き、連続段落を同じ文数・同じ型にそろえない。
   - 「状況→理由→対策」の反復を必要な箇所だけに絞る。
6. **因果に機構を一文添える**
   - 「AだからB」で止めず、なぜAがBを起こすのかを一文で説明する。
   - 根拠のない機構は推測で補わず、確認できない範囲を明記する。

### 3. 話題テストを行う

各文について「読者の理解・判断・行動に必要な状況が更新されたか」を確認する。

- 更新された → 残す
- 「ここから解説します」「詳しく見ていきます」など文書進行の実況だけ → 削る
- 前文の言い換えだけ → 統合する
- 因果や条件が不足 → 確認済みの範囲で補う

### 4. 見出しと句読点を確認する

- 見出しだけで「何が分かるか／何をするか」を推測できるよう、対象・結論・数字のいずれかを加える。
- 「判断基準」「一次情報」「FAQ」など抽象ラベル単独を避ける。
- 文末の丸括弧は `（…）` で閉じ、`（…）。` と句点を重ねない。

### 5. 最小差分で仕上げる

- 良い具体例、実測値、失敗談、固有の言い回しは残す。
- 意味を変えない範囲で文・段落単位に直す。全文を均一な文体へ作り直さない。
- 修正後に、同一文末3連続、体言止めゼロ、段落文数の均質化、空句、話題の非更新、因果の飛躍を再確認する。

## 入れないもの

- 結論遅延や「最後まで読めば分かる」型の引っ張り
- 体言止め45〜75%・60%±5%などの比率目標
- 一文ごとの機械的な改行
- 「です・ます」や「筆者」の一律禁止
- 禁止語リストの丸写しによる単語検閲
- 文体改善を理由にしたCTA・内部リンク・事実関係の無断変更

## 完了報告

次の3点を短く報告する。

1. 改善した観点（文末／拍・密度／空句／根拠表現／話題／因果）
2. 意図的に維持した内容（検索意図、構成、CTAなど）
3. 未確認・任意改善として残した点
